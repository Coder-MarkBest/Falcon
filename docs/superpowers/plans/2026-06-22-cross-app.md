# Cross-App Discovery & Security Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** App A can call `Falcon.getService<INavService>().getRoute()` and reach the implementation in App B, with signature verification and per-package permission control, by declaring `peerPackages("com.oem.nav", ...)` in one line.

**Architecture:** `peerPackages` is a set of package names the client communicates with → `PeerManager` constructs `content://<pkg>.falcon.registry/services` per peer and discovers across all of them → `getService` probes all peers and returns the matching proxy transparently. `IpcRegistryProvider` becomes `exported="true"` with same-UID guard on `insert`. Signature and permission models (existing `SecurityConfig`) need no code changes.

**Tech Stack:** Kotlin, Android ContentProvider/Binder, JUnit4 + Robolectric (JVM), emulator (instrumented test). KSP/generator files are unchanged.

**Key non-change:** The `FalconGeneratedRegistry`, Proxy, Dispatcher, `IpcEnvelope`, `IpcDispatcher`, `EventProxy`, and `IIpcHost` AIDL are untouched. Cross-app changes are purely in the discovery and security wiring layers.

- [ ] **Step 4: Change `connectPeer` to use the correct authority**

`connectPeer` currently calls `context.contentResolver.call(registryUri, "getHost", ...)` with the single URI. It needs to know which URI contains which process name. Pass the URI through:

```kotlin
private fun connectPeer(processName: String) {
    // Find the registry URI that hosts this process by probing each.
    // For simplicity, try each URI until one returns a binder with matching
    // process info. In the common case of one peer per authority, the first
    // call succeeds immediately.
    // Implementation: iterate registryUris, call("getHost"), check.
    ...
}
```

Wait — the old design stored one registry URI per PeerManager, with one URI per process. But now we have N URIs (peer packages). The connectPeer path needs to know WHICH URI to call. 

Better approach: `connectPeer` takes the URI as a parameter. `refreshPeers` discovers (processName, URI) pairs. Let me design this properly.

In `refreshPeers`, change from simply collecting process names to collecting `(processName, registryUri)` pairs:

```kotlin
private fun queryPeers(): Map<String, Uri> {
    val result = mutableMapOf<String, Uri>()
    for (uri in registryUris) {
        val cursor = try {
            context.contentResolver.query(uri, null,
                "process_name != ?",
                arrayOf(ProcessUtils.getCurrentProcessName(context)),
                null)
        } catch (e: Exception) { null } ?: continue
        cursor.use {
            val colIdx = it.getColumnIndex("process_name")
            if (colIdx < 0) continue
            while (it.moveToNext()) {
                val name = it.getString(colIdx)
                if (name !in result) result[name] = uri
            }
        }
    }
    return result
}
```

Then in `refreshPeers`, iterate `queryPeers()` entries and call `connectPeer(name, uri)`:

```kotlin
private fun refreshPeers() {
    threadPool.submit {
        try {
            val peers = queryPeers()
            peers.forEach { (name, _) ->
                if (!connections.containsKey(name) && !reconnecting.contains(name)) {
                    connectPeer(name)
                }
            }
        } catch (e: Exception) {
            FalconLogger.e("Peer", "refreshPeers failed — peer discovery may be stale", e)
            handler.postDelayed({ refreshPeers() }, 30_000L)
        }
    }
}
```

And `connectPeer(processName)` becomes `connectPeer(processName, registryUri)`. Inside, `call(registryUri, "getHost", ...)` uses the passed uri. Continue storing the connection keyed by processName (unique across all peers).

    fun buildQueriesErrorMessage(missingPackages: Set<String>): String = buildString {
        appendLine("Falcon: the following peer packages are not declared in <queries>. ")
        appendLine("Add the following to your AndroidManifest.xml inside the <manifest> element:")
        appendLine()
        appendLine("<queries>")

In `start()`, after building `registryUris`, check queries using `PackageManager.queryIntentActivities` or try-catch the call:

```kotlin
// Quickest reliable check: try a light query on the first peer provider.
// If the provider is not reachable due to missing <queries>,
// SecurityException is thrown with a clear message.
val missingQueries = mutableListOf<String>()
for (pkg in config.peerPackages) {
    val uri = Uri.parse("content://$pkg.falcon.registry/services")
    try {
        // Attempt a metadata-only query; just need to see if it's reachable.
        context.contentResolver.query(uri, null, null, null, null)?.close()
    } catch (e: SecurityException) {
        missingQueries.add(pkg)
    } catch (e: Exception) {
        // Other errors (uninstalled, etc.) — not a queries issue.
        FalconLogger.d("Falcon", "Peer $pkg not yet available: ${e.message}")
    }
}
if (missingQueries.isNotEmpty()) {
    throw IllegalStateException(buildQueriesErrorMessage(missingQueries.toSet()))
}
```
